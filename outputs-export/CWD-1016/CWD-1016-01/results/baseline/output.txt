## Root Cause

The vulnerability is an **out-of-bounds write** (CWE-787). The `ymin` value is read from the EXR file's `dataWindow` attribute via `bytestream2_get_le32(&s->gb)` at line 646 and stored in `s->ymin`. While there are checks that `ymin <= ymax`, there is **no validation that `ymin` is within the bounds of the allocated frame buffer** (which is sized based on `displayWindow`, not `dataWindow`). In `decode_frame`, the loop at lines 947-952 iterates from 0 to `s->ymin`, calling `memset(ptr, 0, out_line_size)` for each iteration. When `ymin` is a large malicious value (e.g., `INT_MAX`), the code writes far beyond the allocated buffer, causing an out-of-bounds write.

## Trigger Path

1. **`bad.c:804` `decode_frame`**
   - Entry point that processes an EXR frame from `AVPacket`. Initializes get byte context and calls `decode_header`.

2. **`bad.c:426` `decode_header`**
   - Parses EXR header attributes. At line 646, reads `ymin` from the stream via `bytestream2_get_le32(&s->gb)` and stores it in `s->ymin`. The validation at lines 650-655 only checks `ymin <= ymax`, not buffer bounds.

3. **`bad.c:804` `decode_frame` (continued)**
   - After header parsing, at line 894 calls `ff_set_dimensions(avctx, s->w, s->h)` which allocates frame buffer based on `displayWindow` dimensions, not `dataWindow`. At line 947-952, iterates `for (y = 0; y < s->ymin; y++)` and calls `memset(ptr, 0, out_line_size)` - this is the sink where the out-of-bounds write occurs.
