## Root Cause

The vulnerability is an out-of-bounds write in the `append_command` function at line 351 of `bad.c`. The function uses a static buffer `IObuff` of size `IOSIZE` (1025 bytes). The vulnerable `STRCAT(IObuff, ": ")` call at line 351 appends to the buffer without checking if there's sufficient space. The issue is triggered when `do_one_cmd` first fills `IObuff` with an error message via `STRCPY(IObuff, _(e_not_an_editor_command))` at line 663, then calls `append_command` which attempts to append ": " to an already-full buffer, causing the out-of-bounds write.

## Trigger Path

1. `bad.c:370` `do_one_cmd`
   - Entry point for processing Ex commands. When `ea.cmdidx == CMD_SIZE` (unknown command), it copies an error message to IObuff then calls append_command.

2. `bad.c:663` (within `do_one_cmd`)
   - Executes `STRCPY(IObuff, _(e_not_an_editor_command))` which fills the IObuff buffer with the error message string.

3. `bad.c:669` or `bad.c:671` or `bad.c:1269` (within `do_one_cmd`)  
   - Calls `append_command(after_modifier)` or `append_command(*cmdlinep)` to append the attempted command to the error message.

4. `bad.c:351` `append_command`
   - Executes the vulnerable `STRCAT(IObuff, ": ")` without checking buffer bounds, causing the out-of-bounds write when IObuff is already full from the prior STRCPY.
