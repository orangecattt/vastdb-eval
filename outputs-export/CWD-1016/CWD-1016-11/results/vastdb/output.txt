## Root Cause

The vulnerability is an out-of-bounds write (CWE-787) in the `vim_regsub_both` function. The static array `eval_result[MAX_REGSUB_NESTING]` has 4 elements (indices 0-3), but the boundary check at line 393 uses `if (nesting == MAX_REGSUB_NESTING)` which allows `nested = nesting` to be set to values 0-3, then increments `nesting` to reach 4. When deeply nested recursive calls occur (e.g., substitute() expressions within substitute() expressions), the index `nested` can become equal to or exceed `MAX_REGSUB_NESTING` (4), causing `eval_result[nested]` to access memory beyond the array bounds. The STRCPY at line 418 then writes to an unallocated/out-of-bounds memory location.

## Trigger Path

1. `bad.c:882` `do_string_sub`
   - Entry point for string substitution, calls vim_regsub at line 939 and 951

2. `bad.c:850` `vim_regsub`
   - Sets up regex match state and calls vim_regsub_both at line 872

3. `bad.c:361` `vim_regsub_both`
   - The vulnerable function. At line 379, `nesting` is a static variable (initialized to 0). At line 398, `nested = nesting` captures the current nesting level. At line 393, the boundary check `nesting == MAX_REGSUB_NESTING` is too permissive (should be `>=`). At line 445, `++nesting` increments before evaluation. When substitute() expressions trigger recursive calls, `nesting` can reach 4, making `nested` equal to or exceed the array bound 3, causing the out-of-bounds write at line 418: `STRCPY(dest, eval_result[nested]);`
