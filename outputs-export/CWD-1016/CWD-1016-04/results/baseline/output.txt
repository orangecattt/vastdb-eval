## Root Cause

The vulnerability is an out-of-bounds write in the `convolution_y` function. The issue stems from `s->width = w;` at line 403 in `ff_vmafmotion_init`, which stores the width from the filter's input link configuration (`ctx->inputs[0]->w`). Later, in `ff_vmafmotion_process` (line 497-498), `convolution_y` is called using `s->width` and `s->height` from the filter context, but the actual frame `ref` may have different dimensions. If `ref->data[0]` has a larger width than the configured `s->width`, the function reads out-of-bounds from `ref->data[0]`. More critically, the destination buffer `s->temp_data` was allocated based on the configured `s->stride * s->height`, but the writes at lines 457, 466, and 479 use `ref->linesize[0]` as the source stride while still using `s->stride` as the destination stride, causing a mismatch that leads to out-of-bounds writes when processing frames with different stride values.

## Trigger Path

1. `bad.c:423` `config_input_ref`
   - Called during filter initialization to configure the filter with input link dimensions. Calls `ff_vmafmotion_init` with `ctx->inputs[0]->w` and `ctx->inputs[0]->h`, storing these in `s->width` and `s->height`, and allocating buffers based on these dimensions.

2. `bad.c:396` `ff_vmafmotion_init`
   - Sets `s->width = w;` at line 403 and allocates `s->temp_data` with size `s->stride * h` where `s->stride = FFALIGN(w * sizeof(uint16_t), 32)`.

3. `bad.c:518` `do_vmafmotion`
   - Called for each frame during filtering. Receives `AVFrame *ref` which may have different dimensions than configured. Calls `ff_vmafmotion_process(&s->data, ref)`.

4. `bad.c:493` `ff_vmafmotion_process`
   - Calls `s->vmafdsp.convolution_y(s->filter, 5, ref->data[0], s->temp_data, s->width, s->height, ref->linesize[0], s->stride)` at lines 497-498. Uses filter context dimensions (`s->width`, `s->height`) rather than actual frame dimensions, with source stride from `ref->linesize[0]` and dest stride from `s->stride`.

5. `bad.c:433-482` `convolution_y_8bit` / `convolution_y_10bit`
   - The sink function that performs out-of-bounds writes. At lines 457, 466, and 479, it writes to `dst[i * dst_stride + j]` using `dst_stride = s->stride` but reads from source using `src_stride = ref->linesize[0] / sizeof(type)`. When frame stride differs from configured stride, buffer overflow occurs.
