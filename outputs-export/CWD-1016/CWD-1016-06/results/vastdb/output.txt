## Root Cause

The vulnerability is an out-of-bounds write in `filter16_complex_low` at line 537. The root cause is that `s->planeheight[3]` is incorrectly set to `inlink->h` at line 555 in `config_input`, even though typical pixel formats (like YUV420) only have 3 planes. When a format with 4 planes (or when the code iterates using the 4-element `planeheight` array) is processed, the `height` value derived from `planeheight[3]` is used to compute iteration bounds in `deinterlace_slice`. This incorrect height propagates to populate `in_lines_cur` pointers, leading to out-of-bounds memory access when `filter16_complex_low` dereferences `*in_lines_cur[0]++` at line 537.

## Trigger Path

1. `bad.c:736` `filter_frame` - Entry point that receives input frames and calls `filter` for deinterlacing
2. `bad.c:767` `filter` - Calls `ctx->internal->execute` to dispatch `deinterlace_slice` for each plane, using `s->planeheight[plane]` to determine job count (line 728)
3. `bad.c:591` `deinterlace_slice` - Uses `s->planeheight[plane]` at line 607 to compute `height`, which determines iteration bounds and populates `in_lines_cur` array (lines 636-645)
4. `bad.c:525` `filter16_complex_low` - Called via function pointer `s->dsp.filter_complex_low` (line 654), receives `in_lines_cur` array
5. `bad.c:537` (sink) - Out-of-bounds write: `*work_line = *in_lines_cur[0]++ * coef[0];` - dereferences pointer based on incorrectly computed buffer offsets
