## Root Cause

The vulnerability stems from an integer underflow when calculating `nb_samples` at line 419: `nb_samples = ff_inlink_queued_samples(ctx->inputs[0]) - s->nb_samples;`. When `ff_inlink_queued_samples(ctx->inputs[0])` is less than `s->nb_samples`, the subtraction underflows (the function returns `int` but internally operates on `uint64_t`). The crossfade sink at lines 448-451 then uses `s->nb_samples` directly without validating that both input[0] and input[1] have sufficient queued samples. When `s->overlap` is true, the code attempts to consume `s->nb_samples` from each input at lines 436 and 442, but if either input has fewer samples than `s->nb_samples`, the buffers allocated for `cf[0]` and `cf[1]` will be smaller than expected. The subsequent `crossfade_samples` call then writes out-of-bounds because it iterates based on `s->nb_samples` regardless of actual buffer sizes.
