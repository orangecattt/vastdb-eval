## Root Cause

The vulnerability is an out-of-bounds write (CWE-787) in the `crossfade_samples` function. At line 419, `nb_samples` is computed as `ff_inlink_queued_samples(ctx->inputs[0]) - s->nb_samples`, representing extra samples beyond what's needed for crossfade. At the sink (lines 448-451), `s->nb_samples` is passed to `crossfade_samples`, which uses it as a loop bound (line 317: `for (i = 0; i < nb_samples; i++)`) to write to output buffers (line 325: `d[i] = s0[i] * gain0 + s1[i] * gain1`). The bug occurs because the else-if condition at line 430 only verifies `input[1]` has enough samples (`>= s->nb_samples`), but does NOT verify `input[0]` has enough before consuming `s->nb_samples` at line 436. When `input[0]` has fewer than `s->nb_samples` samples, the consumed frame `cf[0]` has insufficient data, yet the full `s->nb_samples` is used as the loop bound, causing out-of-bounds writes.

## Trigger Path

1. `bad.c:389` `activate(AVFilterContext *ctx)`
   - Entry point for the audio crossfade filter, called by FFmpeg filter framework when inputs are ready.

2. `bad.c:418` first condition check
   - Checks `ff_inlink_queued_samples(ctx->inputs[0]) > s->nb_samples`. When FALSE (input[0] has ≤ s->nb_samples), execution falls through to else-if.

3. `bad.c:430` else-if condition
   - Checks `ff_inlink_queued_samples(ctx->inputs[1]) >= s->nb_samples` - only validates input[1, not input[0].

4. `bad.c:436` consume input[0]  
   - Calls `ff_inlink_consume_samples(ctx->inputs[0], s->nb_samples, s->nb_samples, &cf[0])` - attempts to consume s->nb_samples from input[0] without verifying it has enough, producing an undersized frame.

5. `bad.c:448` sink call
   - Calls `s->crossfade_samples(out->extended_data, cf[0]->extended_data, cf[1]->extended_data, s->nb_samples, ...)` - passes the full s->nb_samples count to the loop despite cf[0] potentially having fewer samples.
