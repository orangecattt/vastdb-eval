## Root Cause

The vulnerability is in the `dwa_uncompress` function where `dc_count` is read directly from untrusted compressed data at line 1087 (`dc_count = AV_RL64(src + 72)`) without proper bounds validation. This value is then used to calculate `dest_len` (line 1135) which determines the allocation size for `td->dc_data` (line 1141). Later, the code accesses `dc[idx]` at line 1188 using an index derived from `dc_count` via `dc_w * dc_h * j`, where `dc_w = td->xsize >> 3` and `dc_h = td->ysize >> 3`. When `dc_count` exceeds the valid bounds `(6LL * td->xsize * td->ysize + 63) / 64`, the array access at line 1188 becomes an out-of-bounds read, triggering CWE-129.

## Trigger Path

1. `bad.c:570` `decode_frame`
   - Entry point that processes EXR image packets; calls `avctx->execute2` to process blocks in parallel at line 737.

2. `bad.c:756` `decode_block`
   - Called by the threading executor for each scanline/tile; calls `dwa_uncompress` at line 918 when compression is EXR_DWAA or EXR_DWAB.

3. `bad.c:1061` `dwa_uncompress`
   - Reads `dc_count` from compressed data at line 1087, calculates dest_len at line 1135, allocates buffer at line 1141, and later accesses the out-of-bounds index at line 1188:
   ```c
   const int idx = (x >> 3) + (y >> 3) * dc_w + dc_w * dc_h * j;
   uint16_t *dc = (uint16_t *)td->dc_data;
   dc_val.i = half2float(dc[idx], ...);  // Vulnerable array access
   ```
