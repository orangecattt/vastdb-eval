## Root Cause

The vulnerability is a double-free in the `dialog_changed` function. When a buffer has no filename (`empty_bufname = TRUE`), `buf_set_name()` is called which internally invokes `fname_expand()`. This function sets `buf->b_sfname = buf->b_ffname` (line 514) when `b_sfname` is NULL, creating an alias. Then `buf->b_fname = buf->b_sfname` (line 492), so all three pointers reference the same allocated memory. When the write fails and cleanup occurs (lines 661-663), `VIM_CLEAR(buf->b_fname)` frees the memory and sets the pointer to NULL. However, because `buf->b_sfname` was aliased to point to the same address as `buf->b_ffname` (not `b_fname`), it still points to the freed memory. Thus the subsequent `VIM_CLEAR(buf->b_sfname)` attempts to free already-freed memory, causing a double-free.
