## Root Cause

The vulnerability is a **buffer reallocation without pointer update**. In the `START_HANDLE_STREAM` macro (line 544-554), a copy of a `Bit_Chain` struct (`dat1 = *hdl_dat`) is made before calling `obj_flush_hdlstream`. Inside `obj_flush_hdlstream`, `bit_copy_chain` may call `bit_chain_alloc_size` which uses `realloc` to expand the buffer. After the reallocation, `dat->chain` points to **new memory**, but `dat1.chain` still holds the **old pointer**. When `bit_chain_free(&dat1)` is called (line 551), it frees the old pointer that may still be referenced elsewhere. Later operations like `bit_read_RC` (line 424) that access `dat->chain` can read from freed/stale memory.

## Trigger Path

1. **bad.c:544** START_HANDLE_STREAM macro entry
   - `Bit_Chain dat1 = *hdl_dat` copies struct with current chain pointer before flush operations

2. **bad.c:549** `obj_flush_hdlstream(obj, dat, hdl_dat)`  
   - Calls `bit_copy_chain` which triggers `bit_chain_alloc` at line 482

3. **bad.c:482** `bit_chain_alloc` → `bit_chain_alloc_size`
   - Calls `realloc(dat->chain, dat->size + size)` at line 456, potentially moving the buffer

4. **bad.c:551** `bit_chain_free(&dat1)`  
   - Frees the old pointer saved in dat1 before reallocation occurred

5. **bad.c:424** `bit_read_RC` entry (in called function)
   - `byte = dat->chain[dat->byte]` accesses potentially freed heap memory
