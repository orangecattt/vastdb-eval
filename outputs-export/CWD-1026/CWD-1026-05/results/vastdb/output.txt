## Root Cause

The vulnerability is a Use-After-Free (CWE-416) in the `ImagesToBlob` function. The `blob` pointer is first freed by `UnmapBlob` inside `DetachBlob` at line 629 (when the blob is mapped), which returns the now-freed pointer. Then at line 759, `RelinquishMagickMemory(blob)` is called on the same pointer, causing a double-free/use-after-free. The vulnerability occurs because `DetachBlob` frees the memory internally via `UnmapBlob` but still returns the original pointer, which is then incorrectly freed again by the caller.

## Trigger Path

1. `bad.cpp:692` `ImagesToBlob`
   - Entry function that processes images into blob format. Calls `DetachBlob` at line 755 to extract blob data.

2. `bad.cpp:755` (via call) `DetachBlob`
   - Calls `DetachBlob(images->blob)` to detach the blob. This is an external/lib call to the function at line 619.

3. `bad.cpp:629` `DetachBlob` (UnmapBlob call)
   - Inside `DetachBlob`, when the blob is memory-mapped (`blob_info->mapped != MagickFalse`), `UnmapBlob(blob_info->data, blob_info->length)` frees the underlying memory. The function then returns this now-freed pointer to the caller.

4. `bad.cpp:759` `ImagesToBlob` (RelinquishMagickMemory call)
   - After `DetachBlob` returns the freed pointer, the code at line 759 calls `blob=RelinquishMagickMemory(blob)`, attempting to free the same pointer that was already freed by `UnmapBlob` in the previous step. This is the sink: a second free of already-freed memory.
