## Root Cause

The vulnerability is a classic use-after-free. The `apply_relocations` function allocates a symbol table via `GET_ELF_SYMBOLS` (line 487), then frees it at line 583 with `free(symtab)`. However, the `target_specific_reloc_handling` function uses a static variable `saved_sym` (line 314) that stores a pointer into the symtab (line 322: `saved_sym = symtab + get_reloc_symindex(reloc->r_info)`). When later accessing `saved_sym->st_value` at line 347, the symtab memory has already been freed, causing a use-after-free condition. The static variable persists beyond the function call where symtab is valid.

## Trigger Path

1. `bad.c:783` `process_section_contents`
   - Calls `display_debug_section(i, section, file)` which eventually triggers debug section loading

2. `bad.c:733` `display_debug_section`
   - Calls `load_specific_debug_section` to load the debug section

3. `bad.c:679` `load_specific_debug_section`
   - Calls `apply_relocations((FILE *)file, sec, section->start, section->size, &section->reloc_info, &section->num_relocs)` when relocation is needed

4. `bad.c:498` `apply_relocations`
   - Calls `target_specific_reloc_handling(rp, start, symtab)` for each relocation entry. This is where `saved_sym` is set to point into the symtab

5. `bad.c:322` `target_specific_reloc_handling` (EM_MSP430 case)
   - Sets `saved_sym = symtab + get_reloc_symindex(reloc->r_info)` to store a symbol pointer from the symtab

6. `bad.c:583` `apply_relocations`
   - After processing all relocations, calls `free(symtab)` - this frees the memory that `saved_sym` points to

7. `bad.c:347` `target_specific_reloc_handling`
   - Accesses `saved_sym->st_value` after symtab has been freed - **use-after-free trigger**
