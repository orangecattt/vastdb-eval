## Root Cause

The vulnerability is a **Use-After-Free (CWE-416)** in the tuple parsing logic. When `eval_tuple` is called at line 292 in `eval9_nested_expr`, it receives `rettv` as the first tuple element. Inside `eval_tuple` at lines 1332-1336, it copies rettv's value into a newly allocated tuple via `tuple_append_tv` using a shallow copy (`*TUPLE_ITEM(tuple, idx) = *tv`). Both rettv and tuple now point to the same underlying data (e.g., string, list, or nested tuple). When parsing fails (e.g., missing comma at line 1400) and goes to `failret` at line 1410, `tuple_free(tuple)` at line 1412 frees the tuple and all its contents including the data referred to by the original rettv. However, the original caller's `rettv` still contains the freed pointer. Any subsequent use of that `rettv` after line 292 returns FAIL triggers the use-after-free.

## Trigger Path

1. `bad.c:278` `eval9_nested_expr` - parses nested expression starting with '('
2. `bad.c:286` `eval1(arg, rettv, evalarg)` - evaluates first element, result stored in rettv
3. `bad.c:292` `eval_tuple(arg, rettv, evalarg, TRUE)` - called when comma detected; passes rettv as first element to tuple
4. `bad.c:1310` `eval_tuple` - processes tuple element at lines 1332-1336: copies rettv into tuple (shallow copy)
5. `bad.c:1400` - error: missing comma causes failure
6. `bad.c:1412` `tuple_free(tuple)` - frees the tuple including data originally from rettv (the sink)
7. After `eval_tuple` returns FAIL, the caller's rettv still holds freed pointer - any subsequent use triggers UAF
