## Root Cause

The vulnerability is a **Use-After-Free** in the tuple evaluation code. In `eval9_nested_expr` (line 249), the function may call `eval_tuple` twice on the same `rettv` structure: once at line 283 for an empty tuple case, and potentially again at line 292 when a comma is encountered. Inside `eval_tuple` (line 1310), a `tuple_T` is allocated at line 1328, populated with values from `rettv`, and stored back into `rettv` via `rettv_tuple_set` at line 1419. However, on error paths (lines 1410-1412), the tuple is freed via `tuple_free(tuple)`. When the second call to `eval_tuple` occurs on the same `rettv` that was already freed, or when the tuple pointer in `rettv->vval.v_tuple` is accessed after being freed, the program accesses freed memory, triggering CWE-416.

## Trigger Path

1. `bad.c:1278` `eval9`
   - Input `(` triggers case for nested expression, dispatches to `eval9_nested_expr`

2. `bad.c:249` `eval9_nested_expr`
   - After skipping `(` and checking for lambda, parses expression. When `**arg == ','`, calls `eval_tuple(arg, rettv, evalarg, TRUE)` at line 292. The same `rettv` pointer is passed to both calls.

3. `bad.c:1310` `eval_tuple`
   - First call allocates `tuple` via `tuple_alloc()` at line 1328, populates it with values from `rettv` using `tuple_append_tv`, then stores it in `rettv` via `rettv_tuple_set(rettv, tuple)` at line 1419.
   - Source: Line 1412 (`tuple_free(tuple)`) frees the tuple on error paths without clearing `rettv->vval.v_tuple`.
   - Sink: Line 292 calls `eval_tuple` again on the same `rettv` whose internal tuple was freed in a previous call.
