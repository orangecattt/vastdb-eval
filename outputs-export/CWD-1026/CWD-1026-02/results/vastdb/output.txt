## Root Cause

The vulnerability is a Use-After-Free (CWE-416) in the SANM video decoder. The `ctx->stored_frame` buffer is freed by `av_freep(&ctx->stored_frame)` in the `destroy_buffers` function (line 278), which both frees the memory and sets the pointer to NULL. Later, when an FTCH chunk is processed, `process_ftch` (line 459) reads `ctx->stored_frame` at line 461 and dereferences it at line 480 without any NULL check. This can occur when `init_buffers` fails during dimension changes in `process_frame_obj`, triggering `destroy_buffers` to free the buffer, yet the decoder continues processing subsequent frames without properly reinitializing `stored_frame`.

## Trigger Path

1. **`bad.c:512` `decode_frame`**
   - Main entry point for decoding video frames; processes chunks in a while loop starting at line 526

2. **`bad.c:553` FOBJ case in `decode_frame`**
   - When an FOBJ chunk is encountered, calls `process_frame_obj(ctx, &ctx->gb)` to decode the frame object

3. **`bad.c:726-734` `process_frame_obj`**
   - At lines 726-734, if frame dimensions exceed current buffer sizes, calls `init_sizes` then `init_buffers` to reallocate buffers
   - If `init_buffers` fails (returns non-zero), it calls `destroy_buffers(ctx)` which frees `stored_frame` at line 278 and returns error

4. **`bad.c:278` `destroy_buffers`**
   - Frees `ctx->stored_frame` via `av_freep(&ctx->stored_frame)`, setting it to NULL

5. **`bad.c:587-590` FTCH case in `decode_frame`**
   - After the failed FOBJ processing (in a subsequent decoder call), when an FTCH chunk is encountered, calls `process_ftch(ctx, size)`

6. **`bad.c:461` `process_ftch`**
   - Reads `ctx->stored_frame` into local variable `sf` without NULL check: `uint8_t *sf = ctx->stored_frame;` — this is the sink where the use-after-free occurs
