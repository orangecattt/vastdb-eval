## Root Cause

The vulnerability is a Use-After-Free in the XMT (XML-based MPEG-4 text format) parser. The `XMT_ODLink` structure (line 200) contains a pointer `OD *od` to an Object Descriptor. During parsing, descriptors are stored in `parser->od_links` list. The sink at line 1644 accesses `l->od->objectDescriptorID` without verifying the memory is still valid. If the underlying `GF_InitialObjectDescriptor` (iod) is freed (via `gf_odf_del_iod` at line 588) while references in `od_links` still point to freed memory, the subsequent access to `l->od->objectDescriptorID` dereferences freed memory.

## Trigger Path

1. `bad.c:1716` `load_xmt_run`
   - Entry point for XMT scene loading, sets up the parser and calls xmt_resolve_od_links

2. `bad.c:1731` `xmt_resolve_od_links` (called from load_xmt_run)
   - Iterates through parser->od_links and accesses l->od->objectDescriptorID at the sink

3. `bad.c:1644` sink location
   - Accesses `l->od->objectDescriptorID` - vulnerable use-after-free location

Note: The source `gf_free(iod)` is not explicitly present in this reduced test case file. In the full GPAC codebase, the free would occur in `gf_odf_delete_descriptor` -> `gf_odf_del_iod` (line 588), which frees the InitialObjectDescriptor while `od_links` may still hold dangling pointers to the freed memory.
