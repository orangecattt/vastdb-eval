## Root Cause

The vulnerability is CWE-763: Release of Invalid Pointer or Reference caused by an off-by-one error in array initialization. In `AcquireBilateralTLS` (bad.cpp:934-954), the array `weights` is allocated with `number_threads+1` elements (line 943), but the subsequent `memset` at line 946 only zeroes `number_threads` elements, leaving `weights[number_threads]` uninitialized. When any allocation fails in the loop at line 949-951, the function returns early by calling `DestroyBilateralTLS(number_threads,weights)` at line 951. In `DestroyBilateralTLS` (bad.cpp:920-932), the loop at line 927 iterates `i <= number_threads` (i.e., `number_threads+1` iterations), attempting to free all pointers including the uninitialized `weights[number_threads]`. When `RelinquishMagickMemory` is called on this garbage pointer at line 929, it triggers release of an invalid pointer.

## Trigger Path

1. `bad.cpp:956` `BilateralBlurImage`
   - Main entry point. Calls `AcquireBilateralTLS` at line 1006 to allocate thread-local storage weights array.

2. `bad.cpp:934` `AcquireBilateralTLS`
   - Allocates `number_threads+1` pointers at line 943, but memset at line 946 only clears `number_threads` elements. The loop at line 947 iterates `i <= number_threads`, attempting to allocate `weights[number_threads]`. If any allocation fails (line 950), returns early via `DestroyBilateralTLS(number_threads,weights)` at line 951.

3. `bad.cpp:920` `DestroyBilateralTLS`  
   - Called for cleanup when `AcquireBilateralTLS` fails. Loop at line 927 iterates `i <= number_threads` and calls `RelinquishMagickMemory(weights[i])` at line 929 for each index including the uninitialized `weights[number_threads]`, triggering the invalid pointer release.
