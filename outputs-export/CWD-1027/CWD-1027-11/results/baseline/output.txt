## Root Cause

The vulnerability occurs because memory allocated by `png_create_read_struct` (line 582) is not released when the program exits early via `return 99;` (line 964). In the `read_png` function, a png_struct is allocated and stored in `dp->read_pp` (line 582), and later moved to `dp->original_pp` (line 635) after `update_display`. When an error occurs (e.g., from `display_log`), a longjmp transfers control back to `do_test`, which returns the error code to `main`. If the error code exceeds `QUIET`, main returns 99 immediately (line 964), bypassing the cleanup calls `display_clean` (line 990) and `display_destroy` (line 994) that would otherwise free the allocated memory.

## Trigger Path

1. `bad.c:868` `main`
   - Entry point; calls `do_test` with file arguments; returns 99 at line 964 when error code > QUIET, bypassing cleanup

2. `bad.c:850` `do_test`
   - Sets up `setjmp` for error handling at line 853; calls `test_one_file` at line 857; returns error code at line 864 when longjmp occurs

3. `bad.c:698` `test_one_file`
   - Calls `update_display` at line 706 which triggers initial PNG read; calls `read_png` multiple times in the test loop; memory allocated during reads is stored in `dp->original_pp` and `dp->original_ip`

4. `bad.c:564` `read_png`
   - Calls `display_clean_read` at line 574 to clean previous state; allocates png_struct via `png_create_read_struct` at line 582 (source location); allocates png_info via `png_create_info_struct` at line 590

5. `bad.c:947` or `bad.c:964` (sink - `return 99;`)
   - When error handling triggers longjmp, control returns to `do_test` which passes the error code to `main`, causing early return 99 that skips memory cleanup
