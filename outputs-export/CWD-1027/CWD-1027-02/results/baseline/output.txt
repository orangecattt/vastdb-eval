## Root Cause

The vulnerable object is `tile->out_ref` (an `AVFrame*` pointer) allocated at line 230 in `bad.c` via `ff_get_video_buffer()`. This buffer stores the composited video frame output. The `uninit` function (lines 275-279) only frees `tile->prev_out_ref` but **never frees `tile->out_ref`**, causing a memory leak. When `filter_frame` is called for the first time (when `tile->out_ref` is NULL), it allocates a new video buffer that remains allocated for the filter's lifetime but is never released when the filter is destroyed.

## Trigger Path

The trigger path for this FFmpeg tile filter vulnerability is:

1. **External filter graph/API (outside this test case)** - Sets up and configures an FFmpeg filter graph containing the "tile" filter; sends video frames through the filter chain
2. **ff_filter_frame (line 196, declared external)** - Called by external code to push video frames into the filter; dispatches to the filter's `filter_frame` callback
3. **filter_frame (bad.c:222)** - Receives incoming video frames; on first call when `tile->out_ref` is NULL, allocates a new video buffer at line 230
4. **ff_get_video_buffer (bad.c:230)** - Allocates the `out_ref` buffer via `ff_get_video_buffer(outlink, outlink->w, outlink->h)` - this is the vulnerable allocation point
5. **uninit (bad.c:275)** - Called when filter is destroyed/unloaded; fails to free `out_ref` (only frees `prev_out_ref` at line 279), triggering the memory leak

Note: The external FFmpeg framework that initializes and tears down filters is not present in this test case directory but is required to trigger the full path. The key vulnerability manifests when `uninit` is called after `filter_frame` has allocated `out_ref`.
