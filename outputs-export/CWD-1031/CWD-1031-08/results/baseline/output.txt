## Root Cause

The vulnerability is a NULL pointer dereference in `class_object_index` (vim9class.c). When `rettv->v_type` is `VAR_OBJECT`, the code checks if `rettv->vval.v_object` is NULL (lines 506-510) but fails to check if `obj_class` is NULL before dereferencing it. At line 511, `cl = rettv->vval.v_object->obj_class` assigns a potentially NULL `obj_class` pointer to `cl`. Later at line 517, accessing `cl->class_class_function_count` dereferences this NULL pointer, triggering the vulnerability. An object can exist with a NULL `obj_class` pointer, bypassing the existing NULL check.

## Trigger Path

1. `bad.c:818` `eval9`
   - Parses and evaluates expressions, calls `handle_subscript` at line 1088 to handle `.` member access operators

2. `bad.c:633` `handle_subscript`
   - Processes `.` operator for class/object member access at lines 786-799, calls `class_object_index` when `rettv->v_type` is `VAR_CLASS` or `VAR_OBJECT`

3. `bad.c:482` `class_object_index`
   - Receives `rettv` with `v_type == VAR_OBJECT`. At line 506-510, checks `rettv->vval.v_object == NULL` but misses checking `obj_class`. At line 511, `cl = rettv->vval.v_object->obj_class` can receive NULL. At line 517, `cl->class_class_function_count` dereferences NULL.
