## Root Cause

The vulnerability is a NULL pointer dereference (CWE-476) in the `display_relocations` function at bad.c:1318. The `section` parameter is dereferenced to access `section->sh_type` without any NULL check. The vulnerability can be triggered when `filedata->section_headers` is NULL but `filedata->file_header.e_shnum` is non-zero. This occurs when `process_section_headers` fails to load section headers (e.g., due to memory allocation failure or malformed ELF file), and the code continues to process static relocations in the `else` branch of `process_relocs` when `do_using_dynamic` is FALSE.

## Trigger Path

1. `bad.c:1614` `process_object`
   - Entry point that processes an ELF file. Calls `get_section_headers` for a probe read at line 1643.

2. `bad.c:1651-1654` (in `process_object`)
   - Frees and sets `filedata->section_headers` to NULL after probe read to force re-reading full section headers.

3. `bad.c:1656` `process_section_headers`
   - Attempts to load full section headers. Can fail due to invalid/malformed ELF or memory allocation failure, leaving `section_headers` as NULL.

4. `bad.c:1661-1662` (in `process_object`)
   - If `do_using_dynamic` is FALSE and section_headers failed, sets `do_reloc = false`. But if `do_using_dynamic` is TRUE, `do_reloc` remains true.

5. `bad.c:1673` `process_relocs`
   - Called even when section_headers is NULL.

6. `bad.c:1568-1580` (in `process_relocs`, `else` branch)
   - When `do_using_dynamic` is FALSE, iterates through `filedata->section_headers` which is NULL while `e_shnum` may be non-zero from the file header.

7. `bad.c:1578` `display_relocations(section, ...)`
   - Called with potentially NULL `section` pointer.

8. `bad.c:1318` `display_relocations`
   - **Sink**: Accesses `section->sh_type` without NULL check, triggering NULL pointer dereference:
   ```c
   relocation_type rel_type = rel_type_from_sh_type (section->sh_type);
   ```
