## Root Cause

The vulnerability is a CWE-476 NULL Pointer Dereference in the VP3 decoder. At `bad.c:681-682`, the code allocates `edge_emu_buffer` using `av_malloc(9 * FFABS(s->current_frame.f->linesize[0]))` but **never checks if the allocation succeeded** before using the pointer. If `av_malloc` returns NULL (due to memory exhaustion), the code proceeds normally and later dereferences the NULL pointer in `render_slice` at lines 513 and 527, causing a crash.

## Trigger Path

1. `bad.c:591` `vp3_decode_frame`
   - Main decode function, entry point for processing video frames

2. `bad.c:681-682` (inside `vp3_decode_frame`)
   - Allocates `edge_emu_buffer`: `s->edge_emu_buffer = av_malloc(9 * FFABS(s->current_frame.f->linesize[0]));`
   - No NULL check after allocation; if `av_malloc` returns NULL, execution continues

3. `bad.c:800` (inside `vp3_decode_frame`)
   - Calls `render_slice(s, i)` in a loop to render each slice

4. `bad.c:418` `render_slice`
   - Called by `vp3_decode_frame` to decode and render video slices
   - Uses `s->edge_emu_buffer` without any NULL validation

5. `bad.c:527` (inside `render_slice`)
   - **Sink**: `uint8_t *temp = s->edge_emu_buffer;` - dereferences potentially NULL pointer
   - This occurs when `standard_mc` is true and motion vectors reference pixels outside the frame boundaries, triggering the edge emulation code path

6. `bad.c:513` (inside `render_slice`, VP4 path)
   - **Additional sink**: Another use of `s->edge_emu_buffer` without NULL check

The vulnerability is triggered when: (1) `av_malloc` fails due to memory pressure, AND (2) motion compensation code paths require edge emulation (src_x < 0 || src_y < 0 || src_x + 9 >= plane_width || src_y + 9 >= plane_height), causing `render_slice` to access the NULL buffer.
