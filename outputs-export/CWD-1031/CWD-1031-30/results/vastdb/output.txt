## Root Cause

The vulnerability is a Null Pointer Dereference (CWE-476) in `scan_unit_for_symbols`. When processing `DW_TAG_variable` elements, the code allocates a `struct varinfo` using `bfd_zalloc` (line 744), which zeros all struct fields including the `file` pointer. The variable is then added to `unit->variable_table` (line 750) with its `unit_offset` set (line 751). Later, when another variable with a `DW_AT_specification` attribute is processed, the code looks up this previously allocated variable via `lookup_var_by_offset` (line 850-851) and attempts to copy its `file` field using `strdup(spec_var->file)` (line 864). Since `bfd_zalloc` initialized `spec_var->file` to NULL, passing it to `strdup` causes the null pointer dereference.

## Trigger Path

1. `bad.c:295` `find_abstract_instance`
   - Parses DWARF debug info and calls `comp_unit_maybe_decode_line_info` at line 584 when encountering `DW_AT_decl_file` attributes to resolve file names.

2. `bad.c:297` `comp_unit_maybe_decode_line_info`
   - Called to decode line information for a compilation unit. Calls `scan_unit_for_symbols` at line 638 to scan for function and variable symbols.

3. `bad.c:649` `scan_unit_for_symbols`  
   - Scans the DIE tree for symbols. When processing `DW_TAG_variable` entries, allocates `varinfo` structs via `bfd_zalloc` (line 744, the source) and adds them to `unit->variable_table`.

4. `bad.c:744` Allocation (source)
   - `var = (struct varinfo *) bfd_zalloc(abfd, amt)` - allocates and zero-initializes a `varinfo` struct, leaving `file` as NULL.

5. `bad.c:864` Dereference (sink)
   - `var->file = strdup(spec_var->file)` - dereferences `spec_var->file` which is NULL (from zero-initialization), causing the null pointer dereference when passed to `strdup`.
