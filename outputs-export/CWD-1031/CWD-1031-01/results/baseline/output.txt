## Root Cause

The vulnerability is a NULL pointer dereference in `ff_combine_frame` at line 285. The root cause is that `dnxhd_find_frame_end` can return a negative value (via `return dctx->remaining;` at line 359) when `avpriv_dnxhd_get_frame_size` returns a value ≤ 0 and `dnxhd_get_hr_frame_size` also returns ≤ 0. This negative value is then passed as the `next` parameter to `ff_combine_frame`. When `next` is negative (but not `END_NOT_FOUND`), the function proceeds to access `pc->buffer[pc->last_index + next]` at line 285 without ensuring `pc->buffer` has been initialized. If this is the first parse call or `pc->index` is 0, `pc->buffer` remains NULL, causing the NULL pointer dereference.

## Trigger Path

1. `bad.c:392` `dnxhd_parse`
   - Entry point called by the parser. Calls `dnxhd_find_frame_end` to get `next`, then calls `ff_combine_frame` with `next`.

2. `bad.c:317` `dnxhd_find_frame_end`
   - Calls `avpriv_dnxhd_get_frame_size` which may return a value ≤ 0. When the fallback `dnxhd_get_hr_frame_size` also returns ≤ 0, it returns the negative `dctx->remaining` directly at line 359.

3. `bad.c:224` `ff_combine_frame`
   - Receives a negative `next` value (not `END_NOT_FOUND`). Since `next < 0`, the condition at line 245 fails (next != END_NOT_FOUND), and the buffer allocation at lines 246-258 is skipped. The function then reaches line 285 and dereferences `pc->buffer` which may be NULL, triggering CWE-476.
