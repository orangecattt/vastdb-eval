## Root Cause

The vulnerability is a NULL pointer dereference in the binary search logic. In `_bfd_elf_canonicalize_dynamic_reloc` (line 622), `*storage = NULL` sets a NULL sentinel to terminate the relocation array. Later, in `_bfd_elf_get_synthetic_symtab`, this array is used for binary search. At line 849, `p` is initialized to `dynrelbuf[0]`, which will be NULL if there are no relocations (the array only contains the sentinel). When the binary search doesn't find a matching relocation (the `while` loop at lines 852-865 never sets `p` to a matching element), `p` remains pointing to the NULL sentinel. The condition at line 868 checks `got_vma == p->address` first, which dereferences `p` when it points to NULL, causing the null pointer dereference.

## Trigger Path

1. `bad.c:592` `_bfd_elf_canonicalize_dynamic_reloc`
   - Fills `storage` with relocation entries, then sets `*storage = NULL` as sentinel (line 622)

2. `bad.c:671` `_bfd_elf_get_synthetic_symtab`
   - Calls `bfd_canonicalize_dynamic_reloc` which invokes `_bfd_elf_canonicalize_dynamic_reloc`, populating `dynrelbuf` with relocations followed by a NULL sentinel

3. `bad.c:838-868` `_bfd_elf_get_synthetic_symtab` (binary search loop)
   - Performs binary search through `dynrelbuf` to find matching relocation by address. At line 849, `p = dynrelbuf[0]` initializes p to first element, which is NULL when `dynrelcount` is 0. If `got_vma` doesn't match any relocation address, the while loop (852-865) never sets `p` to a matching element, leaving `p` pointing to the NULL sentinel.

4. `bad.c:868` `_bfd_elf_get_synthetic_symtab`
   - The condition `got_vma == p->address` dereferences `p` when it points to NULL, triggering the null pointer dereference.
