## Root Cause

The vulnerability is a NULL pointer dereference (CWE-476) in the IAMF demuxer. The `audio_element->layers` pointer remains uninitialized when `ambisonics_mode > 1` because the `ambisonics_config` function returns early at line 1781 without allocating the `layers` array (the allocation at lines 1792-1796 is only reached when `ambisonics_mode <= 1`). Later, when `iamf_read_header` processes the audio element at line 1891, it accesses `audio_element->layers[0].substream_count`, which dereferences a NULL pointer since `layers` was never allocated.

## Trigger Path

1. `bad.c:1854` `iamf_read_header`
   - Entry point that reads IAMF header, iterates over audio elements, and at line 1891 accesses `audio_element->layers[0]`

2. `bad.c:1860` - Calls `ff_iamfdec_read_descriptors` to parse audio element descriptors

3. `bad.c:1744` in `ff_iamfdec_read_descriptors` - Calls `audio_element_obu` to parse each audio element OBU

4. `bad.c:1424` in `audio_element_obu` - Calls `ambisonics_config` for scene-based audio elements (audio_element_type == 2)

5. `bad.c:1780` in `ambisonics_config` - **Source condition**: When `ambisonics_mode > 1`, returns at line 1781 without initializing `audio_element->layers`

6. `bad.c:1891` in `iamf_read_header` - **Sink**: Accesses `audio_element->layers[0].substream_count` which dereferences NULL when `ambisonics_mode > 1`

VAST DB evidence confirms the functions exist: `iamf_read_header(AVFormatContext *)` at line 1854, `ff_iamfdec_read_descriptors` at line 1708, `audio_element_obu` at line 1277, and `ambisonics_config` at line 864.
