## Root Cause

The vulnerability is a Null Pointer Dereference (CWE-476). The `det_ctx->p` member in the `DetectEngineThreadCtx` structure is set to NULL at line 2699 in function `DetectRulePacketRules`. Later, code in `DetectEntropyDoMatch` (line 1322) accesses `det_ctx->p->flow` without checking if `det_ctx->p` is NULL. When an entropy detection rule matches during content inspection, `FlowVarAddFloat(det_ctx->p->flow, ded->fv_idx, entropy)` is called, which dereferences the NULL pointer if `det_ctx->p` was previously set to NULL.

## Trigger Path

1. `bad.c:3030` `FlowWorker` - Entry point that processes packets in the flow worker thread
2. `bad.c:2896` `Detect` - Main detection entry function for each packet
3. `bad.c:2849` `DetectFlow` - Calls DetectRun when packet has a flow
4. `bad.c:2750` `DetectRun` - High-level detection orchestration; calls DetectRulePacketRules then DetectRunTx
5. `bad.c:2091` `DetectRulePacketRules` - Iterates through transactions; sets `det_ctx->p = NULL` at line 2699
6. `bad.c:1314` `DetectEntropyDoMatch` - Called during content inspection; contains sink at line 1322 that accesses `det_ctx->p->flow` without NULL check

The trigger occurs when `DetectRulePacketRules` processes transactions in a loop, sets `det_ctx->p = NULL` at line 2699 (source), but then either a subsequent loop iteration or a subsequent packet processing re-enters content inspection which calls `DetectEntropyDoMatch` (sink) before `det_ctx->p` is re-assigned to a valid packet pointer. The code does not validate that `det_ctx->p` is non-NULL before dereferencing it at line 1322.
